## 高风险项（需产品决策）

### 1. 后台刷新 — risk.background_refresh_parity
- **问题**：HarmonyOS 后台执行策略与 iOS BGAppRefreshTask 不同，无法保证 15 分钟级定时刷新
- **建议**：以前台刷新为主路径，后台刷新为尽力而为增强，通过 BackgroundRefreshAdapter 隔离

### 2. iCloud 同步 — risk.cloud_sync_no_equivalent
- **问题**：无直接等效的 iCloud KVS，需选择云后端
- **建议**：CloudSyncAdapter 默认为 no-op，本地持久化先行，云同步作为后续版本

### 3. NLP 情绪/实体分析 — risk.nlp_unavailable
- **问题**：iOS NaturalLanguage 提供设备端情绪评分和命名实体识别，HarmonyOS 可能无等效 NLP Kit
- **建议**：准备基于关键词/正则的启发式降级方案，不阻塞信息流展示

### 4. 桌面组件 — risk.widget_architecture
- **问题**：WidgetKit 的 small/medium/large 模式与 HarmonyOS Form Kit 架构差异大
- **建议**：作为独立交付物，先实现一种卡片尺寸，保留共享缓存服务契约

### 5. 本地 API — risk.local_api_store_rejection
- **问题**：本地回环 HTTP 服务器可能不符合应用商店分发要求
- **建议**：确认是否需要保留，如保留则仅限调试构建

| 权限 | 风险等级 | 用户感知 | 触发时机 |

| ohos.permission.LOCATION | 中 | 是 | 用户开启精确位置天气 |

## 关键建议

1. **优先实现低风险能力**：HttpClient、PersistenceStore、ShareAdapter、RssParser — 这些是核心新闻浏览的基础
2. **尽早验证中等风险能力**：LocationAdapter、NotificationAdapter、TextToSpeechAdapter、WebViewAdapter — 需要实际设备测试
3. **高风险能力需要产品决策**：后台刷新、云同步、NLP 分析、桌面组件、本地 API — 在开发前需明确降级方案
4. **所有平台能力通过 Adapter 隔离**：业务逻辑不直接依赖平台 API，通过适配器接口解耦
