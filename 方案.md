# iOS 应用迁移到 HarmonyOS NEXT — 方案

## 目标

把任意 iOS Swift/SwiftUI 工程，通过 4 个 skill 的流水线，迁移成可构建运行的 HarmonyOS NEXT ArkTS/ArkUI 工程。

每个 skill 产出结构化 JSON 文件，下一个 skill 以这些 JSON 为输入。阶段之间通过文件交接，不通过对话上下文交接。

## 各阶段详解

### Stage 1: ios-analyze — iOS 工程分析

**目标**：读 iOS 源码，产出结构化 JSON 规格，后续 skill 不需要再读 iOS 源码。

**输入**：

| 输入 | 说明 |
|---|---|
| iOS 工程目录 | Swift/SwiftUI 源码、Assets、Info.plist、entitlements |

**产出（7 个 JSON + 3 个 MD）**：

```
output_{{PROJECT_NAME}}/ios-analyze/
├── specs/
│   ├── project.json         # 工程基本信息：bundle_id、targets、依赖
│   ├── modules.json         # 模块结构：每个模块的职责、依赖、公开接口
│   ├── features.json        # ★ 功能清单：三级层级，每个 feature 有验收标准
│   ├── functions.json       # 函数索引：每个函数的类型、输入输出、并发模式
│   ├── screens.json         # ★ 页面规格：layout_spec（UI 树）、navigates_to、interactions
│   ├── capabilities.json    # iOS 系统能力：framework、API、权限
│   └── resources.json       # 资源清单：图标、颜色、字体、SF Symbol
├── reports/
│   ├── ios工程分析.md
│   ├── ios功能摘要.md
│   └── ios界面摘要.md
├── screenshots/png/         # 截图（可选）
└── assets/                  # 导出的资源（可选）
```

**核心产物**：

- **features.json**：三级功能结构（如 `新闻浏览 → 首页信息流 → 新闻列表加载与展示`），每个 feature 有稳定 id、source_refs、acceptance 验收标准。后续所有 skill 按 feature id 领取任务。
- **screens.json**：每个页面的 `layout_spec` 是从 SwiftUI View hierarchy 逐层提取的结构化 JSON 树（section → element → font_size/color/spacing/padding/corner_radius），是 harmony-generate 生成 UI 的**主要输入**。每个控件还有 `effect` 字段记录功能效果链路。

**分析范围**：只分析主 App target。排除 Test、Widget、Extension、SnapshotSupport。

---

### Stage 2: platform-adaptation — 平台能力适配

**目标**：把 iOS 事实转换为 HarmonyOS 侧可执行的迁移策略。

**输入**：

| 输入 | 来源 |
|---|---|
| capabilities.json | ios-analyze |
| features.json | ios-analyze |
| modules.json | ios-analyze |
| functions.json | ios-analyze |
| screens.json | ios-analyze |
| platform-capabilities.json | 参考库（可扩展） |

**产出（6 个 JSON + 1 个 MD）**：

```
output_{{PROJECT_NAME}}/platform-adaptation/
├── capability-coverage.json     # 每个 iOS 能力 → HarmonyOS Kit 映射 + 策略
├── feature-adaptation.json      # 每个 feature 的适配方案
├── implementation-guidance.json # platform_modules 边界 + public API 草案
├── interaction-adaptation.json  # ★ 导航地图 + 交互映射 + SF Symbol 映射
├── concurrency-adaptation.json  # 并发模式映射（task_group → Promise.all 等）
├── risks.json                   # 风险清单（含 recommended_action）
└── reports/平台能力适配摘要.md
```

**核心产物**：

- **interaction-adaptation.json** 包含三张表：
  - `navigation_map`：每条 navigates_to 的 Harmony 实现方式（pushUrl / bindSheet 等）
  - `interactions`：每个交互的组件映射（pull_to_refresh → Refresh 等）
  - `symbol_map`：每个 SF Symbol → HarmonyOS 图标/emoji 映射
- **concurrency-adaptation.json**：每个并发函数的映射（task_group → Promise.all 分批，保留 batch_size）

**策略枚举**：`native_equivalent` / `api_replacement` / `product_degradation` / `custom_service` / `unsupported_pending_decision`

---

### Stage 3: harmony-generate — Harmony 工程生成

**目标**：按 JSON 规格生成完整可构建的 HarmonyOS NEXT 工程。

**输入**：

| 输入 | 来源 |
|---|---|
| 7 个 JSON specs | ios-analyze |
| 6 个适配 JSON | platform-adaptation |
| project-template/ | 工程模板 |
| 截图（可选） | ios-analyze |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-generate/
├── harmony模块实现计划.json    # 每个 iOS 模块 → Harmony 文件映射
└── harmony全量实现追踪.md      # 追踪表：iOS 类型/函数 → Harmony 去向 + 状态

{{PROJECT_NAME}}Harmony/         # 完整 HarmonyOS NEXT 工程
└── entry/src/main/ets/
    ├── pages/                   # 每个 screen → 一个页面
    ├── components/              # component_specs 展开
    ├── stores/                  # @ObservedV2 + @Trace 状态管理
    ├── services/                # 网络请求、数据解析、业务逻辑
    ├── platform/                # 平台适配层（通知、定位、TTS、分享等）
    └── models/                  # 数据模型、常量
```

**关键约束（部分，完整规则见 SKILL.md）**：

| 规则 | 说明 |
|---|---|
| ArkTS 严格模式 | 禁止 any/spread/解构/索引访问/barrel export |
| V2 装饰器 | @ComponentV2、@ObservedV2、@Trace、@Local、@Param |
| layout_spec 驱动 | UI 按结构化 JSON 树逐 section 生成，截图是辅助 |
| 功能端到端实现 | 控件 → 存值 → 效果生效，三层必须全做 |
| 真实数据 | 使用与 iOS 相同的 URL，HTTP 错误用 hilog 记录 |
| errorMessage 渲染 | 数据失败时显示错误 + 重试，不允许空白页面 |
| SF Symbol 映射 | 读 symbol_map，图标 + 文字分两行（不拼字符串） |
| 导航完整 | 每条 navigates_to 有路由代码，不跳过中间页面 |
| 并发还原 | task_group → Promise.all 分批，不退化为串行 |
| 空态/错误态 | 每个 states 数组中的状态都有对应 UI |
| 设置效果 | 每个 toggle/picker 沿 iOS 源码追踪消费侧并实现 |

---

### Stage 4: harmony-verify — 迁移验收

**目标**：对照 iOS 分析产物逐维度验证，产出 gaps 清单。

**输入**：

| 输入 | 来源 |
|---|---|
| features.json | ios-analyze |
| screens.json | ios-analyze |
| functions.json | ios-analyze |
| capabilities.json | ios-analyze |
| interaction-adaptation.json | platform-adaptation |
| concurrency-adaptation.json | platform-adaptation |
| implementation-guidance.json | platform-adaptation |
| Harmony 工程 | harmony-generate |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-verify/
├── verify-report.json    # 10 个维度的逐项验证结果
├── gaps.json             # 缺口清单（含 suggested_fix）
└── reports/迁移验收摘要.md
```

**10 个验证维度**：

| # | 维度 | 检查内容 | 通过标准 |
|---|---|---|---|
| 1 | 构建验证 | HAP 能否构建成功 | 必须 100% |
| 2 | 功能覆盖 | 每个 feature 有实现文件和实质代码 | ≥ 95% |
| 3 | 导航可达 | 每条 navigates_to 有路由代码 | 100% |
| 4 | 交互还原 | 每个 interaction 有 ArkUI 实现 | ≥ 90% |
| 5 | 并发还原 | task_group 不退化为串行 | 100% |
| 6 | 真实数据 | URL 一致、有错误处理、有权限 | 100% |
| 7 | 平台能力 | 每个 platform module 文件存在且有 API | ≥ 95% |
| 8 | layout_spec 还原 | 每个 section 有代码、tab 有图标、picker 有 Select | ≥ 90% |
| 8b | 功能效果链路 | 控件操作后 App 行为会变 | 100% |
| 9 | 视觉还原（可选） | 截图对比 | 加分项 |

**反馈循环**：gaps.json 喂回 harmony-generate 修复，修完再 verify，直到 gaps 清零。

```
harmony-generate ──→ harmony-verify ──→ gaps.json
      ↑                                        │
      └────────── 修复 gaps ←──────────────────┘
```
