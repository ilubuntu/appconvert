# iOS 应用迁移到 HarmonyOS NEXT — 方案

## 目标

把任意 iOS Swift/SwiftUI 工程，通过 4 个 skill 的流水线，迁移成可构建运行的 HarmonyOS NEXT ArkTS/ArkUI 工程。

每个 skill 产出结构化文件，下一个 skill 通过文件交接，不通过对话上下文交接。`ios-analyze` 的 SwiftSyntax、运行时 UI 树和截图都只是内部工作证据，必须合并进同一份 `ios-spec.json.screens`，不能让下游再读取零散中间目录补信息。

`ios-analyze` 对外提供两类输入：`ios-spec.json` 是精简机器规格和唯一事实源；`ios-report.md` 是辅助 LLM 建立产品上下文的报告。报告和 spec 冲突时以 spec 为准，报告中有但 spec 中没有稳定 id 的内容视为分析缺口。

## 各阶段详解

### Stage 1: ios-analyze — iOS 工程分析

**目标**：读 iOS 源码，产出结构化 JSON 规格，使下游阶段不需要再读 iOS 源码。

**输入**：

| 输入 | 说明 |
|---|---|
| iOS 工程目录 | Swift/SwiftUI 源码、Assets、Info.plist、entitlements |

**公共产出（1 个精简 JSON + 1 份辅助报告）**：

```
output_{{PROJECT_NAME}}/ios-analyze/
├── ios-spec.json            # project/modules/features/functions/screens/capabilities/resources
├── ios-report.md              # 辅助下游 LLM 建立上下文，不作为事实源
└── assets/                  # 导出的资源（可选）
```

内部工作文件如 `scan/swiftui_view_tree.json`、`runtime-ui/*.json`、`screenshots/png/*.png` 可以存在，但不属于跨阶段输入。`ios-analyze` 必须把可用信息写回 `ios-spec.json.screens` 的 `layout_spec`、`source_evidence`、`runtime_ui_tree_evidence`、实际存在的 `screenshot_evidence` 和 `visual_notes`。

**内部处理顺序**：

```text
iOS 工程源码
  ↓
程序化证据采集
  - 工程结构 / target / 入口 / 依赖
  - source_index / imports / plist / entitlements
  - SwiftSyntax View Tree
  - asset index / SF Symbol / 颜色
  - 尝试 runtime UI tree，失败写入 ios-spec.json.screens
  - screenshots 为按需证据，用于视觉复刻或视觉比对
  ↓
LLM 读取证据 + 关键源码
  ↓
生成 ios-spec.json
  ↓
生成 ios-report.md
```

原则：SwiftSyntax 和运行时 UI 树是 LLM 分析前的证据输入；截图在视觉复刻任务中作为前置证据输入。LLM 的职责是综合已采集证据和源码，生成同一组 JSON specs。

**核心产物详解**：

#### ★ ios-spec.json.features — 功能清单（最重要的产物）

下游阶段按 feature id 领取和追踪任务。缺了它，G 不知道实现什么，V 不知道验证什么。

feature 粒度按“可独立实现、可独立验收、可独立追踪”划分。搜索、过滤、新增、编辑、删除、导入、导出、清空、通知设置、生物识别、分享等独立用户目标必须拆成单独 feature；一个 Tab、一个设置页或一个模块不能直接合并成一个大 feature。中小型多 Tab 应用通常应输出 15-40 个 feature。

| 字段 | 说明 |
|------|------|
| `features[].id` | 稳定功能 id（如 `feat-primary-list`），全流程引用的锚点 |
| `features[].title` | 功能短标题，给人和模型快速识别任务 |
| `features[].level1/level2/level3` | 三级功能层级，只做分类和报告表格 |
| `features[].summary` | 一句话说明功能目标 |
| `features[].user_flow[]` | 用户入口、主要状态和操作结果 |
| `features[].behavior[]` | 业务规则、排序、分组、过滤、校验、点击、删除、编辑等 iOS 功能事实 |
| `features[].states[]` | 功能状态：`loading` / `populated` / `empty` / `error` / `no_results` 等 |
| `features[].data_sources[].type` | 数据源类型：`network` / `local` / `database` / `system` / `fixed_sample` |
| `features[].data_sources[].url` | 真实 URL（network 类型时必填），G 必须用此 URL 发请求 |
| `features[].acceptance[].description` | 自然语言验收标准，V 逐条检查是否实现 |
| `features[].migration_priority` | `high` / `medium` / `low`，决定实现顺序 |
| `features[].screens/modules/functions/capabilities` | 关联页面、模块、函数和系统能力 id；源码位置通过这些 id 追溯 |
| `features[].source_refs[]` | 直接源码证据，供后续回查 |

→ **P** 按 feature 组织适配策略；**G** 按 feature id 领取实现任务；**V** 按 acceptance 逐条验证。

#### ★ ios-spec.json.screens — 页面规格（UI 生成的主要输入）

UI 信息来源优先级：

1. `SwiftSyntax` 解析 SwiftUI body，作为布局结构主证据，结果折叠进 `layout_spec` 和 `source_evidence`。
2. `XCUITest Accessibility Tree` 必须尝试读取模拟器运行时 UI 树，确认真实渲染出的文本、按钮、Tab、输入框、frame、enabled/selected 等结构化 UI 状态，结果或失败原因折叠进 `runtime_ui_tree_evidence`。
3. 源码核对跨文件状态、数据流、导航目标和控件 effect。
4. iOS 模拟器截图用于视觉复刻或视觉比对任务，补充真实视觉效果和运行态分支，结果或失败原因折叠进 `screenshot_evidence` 和 `visual_notes`。

SwiftSyntax 负责回答“源码里写了什么结构”，XCUITest UI 树负责回答“运行时实际渲染了哪些可访问元素”，截图负责回答“实际看起来是什么样”。模型负责把已采集证据归纳成可消费的同一份 `ios-spec.json.screens`。运行时 UI 树和截图都不能单独决定功能边界、数据来源、模块职责或平台能力。

| 字段 | 说明 |
|------|------|
| `screens[].layout_confidence` | UI 置信度：`high` / `medium` / `low` |
| `screens[].source_evidence[]` | SwiftSyntax / 源码证据，记录 View、文件和节点路径 |
| `screens[].runtime_ui_tree_evidence[]` | XCUITest Accessibility Tree 证据，记录运行时元素、frame、label、identifier、状态 |
| `screens[].screenshot_evidence[]` | 截图证据，记录截图路径、状态、补充的视觉信息和差异 |
| `screens[].visual_notes[]` | 字体、颜色、间距、图片裁剪、空态/有数据态等运行态视觉补充 |
| `screens[].layout_spec.container` | 容器层级，如 `NavigationStack > ScrollView > VStack` → ArkUI `Navigation > Scroll > Column` |
| `screens[].layout_spec.sections[]` | 页面分区，必须结构化记录 `type/elements/row/actions/data_binding/empty_state_ref` |
| `screens[].layout_spec.sections[].row.elements[]` | 列表/网格行元素，记录 `role/binding/resource_ref/font/color/spacing` 等 |
| `screens[].layout_spec.sections[].row.actions[]` | 行级动作，如 `tap`、`swipe_delete`、`context_menu` |
| `screens[].layout_spec.sections[].elements[].effect` | **功能效果链路**：记录控件值被消费的位置和产生的行为变化 |
| `screens[].layout_spec.navigation_bar.trailing_items[]` | 工具栏按钮（如 info / + 按钮），每个必须有 action 或 navigates_to |
| `screens[].layout_spec.tab_bar.items[]` | Tab 栏定义（label + icon + target_screen），仅在根 TabView 页面出现 |
| `screens[].navigates_to[]` | 导航出口数组：target + trigger + style（navigation_push / sheet / tab_switch） |
| `screens[].interactions[]` | 交互数组：pull_to_refresh / swipe_action / toolbar / alert 等 |

→ **P** 从 navigates_to 聚合 navigation_map，从 interactions 提取交互映射；**G** 按 layout_spec 逐 section 生成 ArkUI 代码；**V** 逐 section 检查还原度。

XCUITest UI 树主要补充 SwiftSyntax 不擅长的运行态结构：

- 元素是否真实渲染出来，页面是否空白。
- 文本、按钮、Tab、输入框、列表项的运行时 label / value / identifier。
- 元素 frame、enabled、selected、存在性和可点击性。
- 条件分支实际走向，例如 `if items.isEmpty` 显示空态还是列表。
- 缺少 accessibilityIdentifier 时的稳定定位风险。

截图主要补充 SwiftSyntax 和 UI 树都不擅长的视觉信息：

- 真实字体层级、颜色、背景、圆角、阴影、间距。
- 图片资源是否正确显示、裁剪方式是否符合预期。
- 空态、有数据态、表单态、弹窗态、sheet 态。
- 系统控件真实样式，如 Tab、NavigationBar、Picker、Toggle。

如果 SwiftSyntax、运行时 UI 树和截图不一致，`ios-spec.json.screens` 必须同时保留三种证据：以源码结构作为功能事实，以 UI 树作为运行态结构事实，以截图作为视觉事实，并在 `visual_notes` 中标记差异。

#### ios-spec.json.functions — 函数索引

| 字段 | 说明 |
|------|------|
| `items[].kind` | `model` / `service` / `store` / `view_model` / `utility`，决定 Harmony 侧文件归属 |
| `items[].migration_action` | `model` / `service` / `store` / `arkui_component` / `platform_adapter` / `merge` / `delete_with_reason` |
| `items[].concurrency` | 并发模式枚举（11 种），G 查 platform-capabilities.json 映射为 ArkTS 并发原语 |
| `items[].concurrency_detail` | 并发参数（如 `maxConcurrency=5`、`interval=1s`、`delay=5s`） |
| `items[].type_definition` | 仅 `kind=model` 时出现：列出 enum cases（含 rawValue、显示名）或 struct fields |

→ **P** 提取 concurrency 做能力匹配；**G** 按 migration_action 决定文件归属，按 type_definition 生成模型；**V** 检查并发模式是否正确实现。

#### ios-spec.json.capabilities — iOS 系统能力

| 字段 | 说明 |
|------|------|
| `capabilities[].id` | 能力 id（如 `auth.localauthentication`、`notification.usernotifications`） |
| `capabilities[].category` | 能力分类：`notification` / `auth` / `storage` / `cloud` / `payment` / `share` 等 |
| `capabilities[].ios_framework` | Apple framework 名称 |
| `capabilities[].ios_apis[]` | 关键 API / 类型名数组 |
| `capabilities[].usage` | 当前工程中这个能力的用途 |
| `capabilities[].permission_or_entitlement[]` | 所需权限、Info.plist key 或 entitlement |
| `capabilities[].migration_risk` | `low` / `medium` / `high` |
| `capabilities[].fallback_required` | 是否需要降级或外部服务决策 |

→ **P** 以此为输入做能力映射；**V** 检查每个能力是否有对应 Harmony 实现。

#### ios-spec.json.modules — 模块结构

| 字段 | 说明 |
|------|------|
| `items[].id` | 模块 id（如 `views.method`） |
| `items[].ios_paths[]` | iOS 源码目录/文件 |
| `items[].public_interfaces[]` | 对外接口或主要类型 |
| `items[].depends_on[]` | 模块依赖关系 |
| `items[].suggested_harmony_boundary` | 建议的 Harmony 侧文件边界 |

→ **G** 按模块拆分 Harmony 文件结构；**P** 按模块边界组织适配策略。

#### ios-spec.json.resources — 资源清单

| 字段 | 说明 |
|------|------|
| `colors` | 颜色资源 map，key 是稳定资源 id，value 含 `ios_name`、`usage`、`value/value_light/value_dark`、`source_ref`、`target_name` |
| `symbols` | SF Symbol 资源 map，key 是稳定资源 id，value 含 `ios_name`、`usage`、`source_ref`、`target_name` |
| `images` | 图片资源 map，key 是稳定资源 id，value 含 `ios_name`、`usage`、`source_ref`、`asset_path`、`target_name` |

→ **P** 从 sf_symbol 类型提取 symbol_map；**G** 按颜色和图标资源生成 ArkUI 资源文件。

#### ios-spec.json.project — 工程元数据

| 字段 | 说明 |
|------|------|
| `bundle_identifier` | iOS bundle id → Harmony bundleName |
| `app_entry.file` / `app_entry.type` | 启动入口文件和类型 |
| `entitlements.capabilities[]` | entitlement 列表 |
| `imports[]` | 所有 import 的 framework（含第三方） |

→ **G** 用 bundle_identifier 设置 Harmony 工程配置。

**分析范围**：只分析主 App target。排除 Test、Widget、Extension、SnapshotSupport。

### Stage 2: platform-adaptation — 平台能力适配

**目标**：把 iOS 事实转换为 HarmonyOS 侧可执行的迁移策略。

**输入**：

| 输入 | 来源 |
|---|---|
| ios-spec.json | ios-analyze |
| ios-report.md | ios-analyze（辅助输入，不能覆盖 spec） |
| platform-capabilities.json | 参考库入口 index |
| system-capabilities.json | 系统能力参考 |
| ui-interactions.json | UI 交互/控件适配参考 |
| concurrency-patterns.json | 并发/异步模型参考 |
| data-storage.json | 存储/同步参考 |
| media-capabilities.json | 媒体能力参考 |

**产物（1 个 JSON）**：

```
output_{{PROJECT_NAME}}/platform-adaptation/
└── platform-adaptation.json                         # feature_adaptation / implementation_guidance / interaction_adaptation
```

**核心产物详解**：

#### 分类到输出的映射

| reference 分类 | 输入事实 | 输出位置 | 说明 |
|---|---|---|---|
| `system-capabilities` | `ios-spec.json.capabilities` | `feature_adaptation`、`implementation_guidance.platform_modules` | 真正的平台 Kit、权限、manifest、adapter |
| `data-storage` | capabilities + functions/stores | `feature_adaptation`、`store_requirements` | 本地存储、共享存储、云同步、状态持久化 |
| `media-capabilities` | capabilities | `feature_adaptation`、`platform_modules` | TTS、音频、相机、相册、图片等 |
| `ui-interactions` | screens/interactions/navigates_to/tab_bar | `interaction_adaptation` | Tab、toolbar、sheet、refresh、search、swipe、导航、图标 |
| `concurrency-patterns` | functions.concurrency | `implementation_guidance.concurrency_requirements` | 不是平台能力，不进入 `feature_adaptation` |

#### platform-adaptation.json

```json
{
  "feature_adaptation": {},
  "implementation_guidance": {
    "platform_modules": [],
    "manifest_requirements": [],
    "permission_requirements": [],
    "service_requirements": [],
    "store_requirements": [],
    "concurrency_requirements": []
  },
  "interaction_adaptation": {
    "interactions": [],
    "navigation_map": [],
    "symbol_map": []
  }
}
```

→ **G** 按 `feature_adaptation` 做系统能力/存储/媒体适配，按 `interaction_adaptation` 生成 UI 交互和导航，按 `concurrency_requirements` 还原异步模型。

**策略枚举**：`native_equivalent` / `api_replacement` / `product_degradation` / `custom_service` / `unsupported_pending_decision`

### Stage 3: harmony-generate — Harmony 工程生成

**目标**：按 JSON 规格生成完整可构建的 HarmonyOS NEXT 工程。

**输入**：

| 输入 | 来源 |
|---|---|
| ios-spec.json | ios-analyze |
| ios-report.md | ios-analyze（辅助输入，不能扩大实现范围） |
| platform-adaptation.json | platform-adaptation |
| project-template/ | 工程模板 |
| `ios-spec.json.screens` 中的 `layout_spec` / `runtime_ui_tree_evidence` / `screenshot_evidence` / `visual_notes` | ios-analyze |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-generate/
└── harmony-implementation.json       # module_plan / traceability / build_summary / external_requirements

{{PROJECT_NAME}}Harmony/         # 完整 HarmonyOS NEXT 工程
└── entry/src/main/ets/
    ├── pages/                   # 每个 screen → 一个页面
    ├── components/              # component_specs 展开
    ├── stores/                  # @ObservedV2 + @Trace 状态管理
    ├── services/                # 网络请求、数据解析、业务逻辑
    ├── platform/                # 平台适配层（通知、定位、TTS、分享等）
    └── models/                  # 数据模型、常量
```

#### harmony-implementation.json.module_plan — 模块实现计划

| 字段 | 说明 |
|------|------|
| `modules[].ios_module_id` | iOS 模块 id（对应 ios-spec.json.modules） |
| `modules[].harmony_files[]` | 该模块产出的 Harmony 文件列表 |
| `modules[].ios_types[]` | 该模块包含的 iOS 类型/函数 |
| `modules[].migration_action` | 迁移动作（model / service / store / merge / delete） |
| `modules[].status` | 实现状态（pending / in_progress / completed） |

→ **V** 按此文件检查每个模块是否都有对应 Harmony 文件。

#### Harmony 工程目录职责

| 目录 | 来源 | 规则 |
|------|------|------|
| `pages/` | ios-spec.json.screens 每个 screen → 一个 `*Page.ets` | 页面按 layout_spec 生成，用 Navigation/router 管理导航 |
| `components/` | ios-spec.json.screens 的 component_specs | 可复用子组件，用 `@ComponentV2` + `@Builder` |
| `stores/` | ios-spec.json.functions `kind=store` 或 `view_model` | `@ObservedV2` + `@Trace`，单例模式 |
| `services/` | ios-spec.json.functions `kind=service` + implementation-guidance 的 service_requirements | 网络请求、数据持久化、业务逻辑 |
| `platform/` | implementation-guidance 的 platform_modules | 平台适配层（权限、通知、定位等） |
| `models/` | ios-spec.json.functions `kind=model` | 数据模型，含 enum type_definition 展开 |

**关键约束（部分，完整规则见 SKILL.md）**：

| 规则 | 说明 |
|---|---|
| ArkTS 严格模式 | 禁止 any/spread/解构/索引访问/barrel export |
| V2 装饰器 | @ComponentV2、@ObservedV2、@Trace、@Local、@Param |
| layout_spec 驱动 | UI 按 SwiftSyntax View Tree 整理出的结构化 JSON 树逐 section 生成，视觉补充来自 `ios-spec.json.screens.screenshot_evidence` / `visual_notes` |
| 消费 feature-adaptation | 每个 feature 实现前读 generation_tasks[] + ui_constraints[]；risk_level=high 的 fallback 不允许跳过 |
| 功能端到端实现 | 控件 → 存值 → 效果生效，三层必须全做 |
| 真实数据 | 使用与 iOS 相同的 URL，HTTP 错误用 hilog 记录 |
| errorMessage 渲染 | 数据失败时显示错误 + 重试，不允许空白页面 |
| SF Symbol 映射 | 读 symbol_map，图标 + 文字分两行（不拼字符串） |
| 导航完整 | 每条 navigates_to 有路由代码，不跳过中间页面 |
| 并发还原 | 读 ios-spec.json.functions concurrency 字段 + 查 platform-capabilities.json，按映射实现（timer→setInterval、dispatch_async→setTimeout 等） |
| 空态/错误态 | 每个 states 数组中的状态都有对应 UI |
| 设置效果 | 每个 toggle/picker 沿 iOS 源码追踪消费侧并实现 |

### Stage 4: harmony-verify — 迁移验收

**目标**：对照 iOS 分析产物逐维度验证，产出 gaps 清单。

**输入**：

| 输入 | 来源 |
|---|---|
| ios-spec.json | ios-analyze |
| ios-report.md | ios-analyze（辅助输入，只用于理解主流程和风险） |
| platform-adaptation.json | platform-adaptation |
| harmony-implementation.json | harmony-generate |
| Harmony 工程 | harmony-generate |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-verify/
└── verification.json            # summary / dimensions / gaps
```

#### verification.json — 验证结果

| 字段 | 说明 |
|------|------|
| `build.status` | 构建是否通过（passed / failed） |
| `summary.overall` | 总体结论：`passed` / `passed_with_gaps` / `failed` |
| `summary.total_features` / `passed_features` | 功能覆盖统计 |
| `features[].feature_id` / `status` / `acceptance_results[]` | 每个 feature 的逐条验收结果 |
| `navigations[]` / `interactions[]` / `concurrency[]` | 各维度逐项检查结果 |
| `adaptation_coverage[]` | 维度 7b：每个 adaptation_item 是否实现 + high risk fallback 是否落实 |

#### verification.json.gaps[] — 缺口清单（喂回 G 修复）

| 字段 | 说明 |
|------|------|
| `gaps[].id` | gap 编号 |
| `gaps[].dimension` | 所属维度（feature_coverage / navigation / risk 等） |
| `gaps[].severity` | `critical` / `high` / `medium` / `low` |
| `gaps[].description` | 缺口描述 |
| `gaps[].ios_evidence` | iOS 侧的期望（来自 specs） |
| `gaps[].harmony_evidence` | Harmony 侧的现状（来自源码） |
| `gaps[].suggested_fix` | **具体修复方案**（含文件路径和代码改动建议） |

→ **G** 读 verification.json.gaps 逐条修复，修完重新构建 + verify，直到 gaps 清零或仅剩 low 级。

**验证维度**：

| # | 维度 | 检查内容 | 通过标准 |
|---|---|---|---|
| 1 | 构建验证 | HAP 能否构建成功 | 必须 100% |
| 2 | 功能覆盖 | 每个 feature 有实现文件和实质代码 | ≥ 95% |
| 3 | 导航可达 | 每条 navigates_to 有路由代码 | 100% |
| 4 | 交互还原 | 每个 interaction 有 ArkUI 实现 | ≥ 90% |
| 5 | 并发还原 | ios-spec.json.functions 中标注的并发模式在代码中正确实现 | 100% |
| 6 | 真实数据 | URL 一致、有错误处理、有权限 | 100% |
| 7 | 平台能力 | 每个 platform module 文件存在且有 API + verification[] | ≥ 95% |
| 7b | 适配覆盖与风险缓解 | feature-adaptation 每个 adaptation_item 有实现；risk_level=high 的 fallback 已落实 | ≥ 95% |
| 8 | layout_spec 还原 | 每个 section 有代码、tab 有图标、picker 有 Select | ≥ 90% |
| 8a | 运行时 UI 树对齐 | XCUITest UI 树中的主要 text/button/tab/input/cell 在 Harmony 中可见且状态一致 | ≥ 90% |
| 8b | 快捷操作入口可达 | 每个 toolbar/导航按钮有对应 onClick 实现 | 100% |
| 8c | 功能效果链路 | 控件操作后 App 行为会变（toggle/picker/button → 效果） | 100% |
| 9 | 视觉还原 | `ios-spec.json.screens.screenshot_evidence` / `visual_notes` 与 layout_spec 对照 | 有截图证据时必须核对，缺失时标记视觉证据不足 |

**反馈循环**：verification.json.gaps 喂回 harmony-generate 修复，修完再 verify，直到 gaps 清零。

```
harmony-generate ──→ harmony-verify ──→ verification.json.gaps
      ↑                                        │
      └────────── 修复 gaps ←──────────────────┘
```

## 可选编排模式：多 Agent

当工程规模较大、上下文压力明显或需要并行实现时，可以把 4 个 skill 映射为独立 agent。每个阶段由独立 agent 承载，agent 之间只通过 JSON 文件交接。

**核心收益**：

| 收益 | 说明 |
|------|------|
| 强制产物契约 | Agent 只能看到显式传入的文件列表，无法偷看原始 iOS 源码，强迫只消费结构化 JSON |
| 隔离失败 + 定点重试 | verify 报 N 个 gap，只需重跑 G agent 修 gap，不重跑 ios-analyze 和 platform-adaptation |
| 并行加速 | harmony-generate 可按 feature 分拆到多个 agent 并行生成；ios-analyze 的各 step 也可并行 |
| 阶段间质量门 | Agent 之间插入 schema 校验脚本，ios-spec.json.features 不合格直接重跑，不等 G 才发现问题 |
| Token 预算隔离 | 每个阶段独立上下文空间，ios-analyze 读 40 个 Swift 文件不影响 G 的可用空间 |
| 差异化模型 | 分析阶段用强推理模型，生成阶段用执行型模型，控制成本 |

**架构示意**：

```
                    Agent 编排层
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
  ios-analyze     platform-        harmony-generate
    agent          adaptation         agent
        │           agent         ┌────┴────┐
        │               │        ▼         ▼
        ▼               ▼    feature-A   feature-B
   schema 校验      schema 校验   agent       agent
      (脚本)          (脚本)       │           │
        │               │         └────┬──────┘
        ▼               ▼              ▼
                   harmony-verify-agent
                          │
                     verification.json.gaps
                          │
                   ┌──────┴──────┐
                   ▼             ▼
              gap < 阈值?    重跑 G agent
                   │           修复 gaps
                   ▼
                 完成
```

**切换条件**：当 skill 模式在 3+ 个不同工程上验证产物质量稳定后，切换为多 Agent 模式。
