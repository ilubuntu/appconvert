# iOS 应用迁移到 HarmonyOS NEXT — 方案

## 目标

把任意 iOS Swift/SwiftUI 工程，通过 4 个 skill 的流水线，迁移成可构建运行的 HarmonyOS NEXT ArkTS/ArkUI 工程。

每个 skill 产出结构化 JSON 文件，下一个 skill 以这些 JSON 为输入。阶段之间通过文件交接，不通过对话上下文交接。`ios-analyze` 的 SwiftSyntax、运行时 UI 树和截图都只是内部工作证据，必须合并进同一份 `specs/screens.json`，不能让下游再读取零散中间目录补信息。

## 各阶段详解

### Stage 1: ios-analyze — iOS 工程分析

**目标**：读 iOS 源码，产出结构化 JSON 规格，后续 skill 不需要再读 iOS 源码。

**输入**：

| 输入 | 说明 |
|---|---|
| iOS 工程目录 | Swift/SwiftUI 源码、Assets、Info.plist、entitlements |

**公共产出（7 个 JSON + 1 个 MD）**：

```
output_{{PROJECT_NAME}}/ios-analyze/
├── specs/
│   ├── project.json         # 工程基本信息：bundle_id、targets、依赖
│   ├── modules.json         # 模块结构：每个模块的职责、依赖、公开接口
│   ├── features.json        # ★ 功能清单：三级层级，每个 feature 有验收标准
│   ├── functions.json       # 函数索引：每个函数的类型、输入输出、并发模式
│   ├── screens.json         # ★ 页面规格：layout_spec（UI 树）、navigates_to、interactions
│   ├── capabilities.json    # iOS 系统能力：framework、API、权限
│   └── resources.json       # 资源清单：图标、颜色、字体、SF Symbol
├── reports/
│   └── ios分析摘要.md
└── assets/                  # 导出的资源（可选）
```

内部工作文件如 `scan/swiftui_view_tree.json`、`runtime-ui/*.json`、`screenshots/png/*.png` 可以存在，但它们不是后续 skill 的输入。`ios-analyze` 必须把可用信息写回 `screens.json` 的 `layout_spec`、`source_evidence`、`runtime_ui_tree_evidence`、`screenshot_evidence` 和 `visual_notes`。`reports/ios分析摘要.md` 只给人快速浏览，不给下游 skill 使用。

**内部处理顺序**：

```text
iOS 工程源码
  ↓
程序化证据采集
  - 工程结构 / target / 入口 / 依赖
  - source_index / imports / plist / entitlements
  - SwiftSyntax View Tree
  - asset index / SF Symbol / 颜色
  - 尝试 runtime UI tree，失败写入 screens.json
  - screenshots 能力保留，默认不执行，后续按需开启
  ↓
LLM 读取证据 + 关键源码
  ↓
生成 specs/*.json
  ↓
生成 reports/ios分析摘要.md
```

原则：SwiftSyntax 和运行时 UI 树是 LLM 分析前的证据输入；截图能力保留，若开启也作为前置证据输入。LLM 的职责是综合已采集证据和源码，生成同一组 JSON specs。

**核心产物详解**：

#### ★ features.json — 功能清单（最重要的产物）

后续所有 skill 按 feature id 领取和追踪任务。缺了它，G 不知道实现什么，V 不知道验证什么。

| 字段 | 说明 |
|------|------|
| `items[].id` | 稳定功能 id（如 `feat-brew-calculator`），全流程引用的锚点 |
| `items[].level1/level2/level3` | 三级功能层级，如 `用户中心 → 个人设置 → 切换深色模式` |
| `items[].data_sources[].type` | 数据源类型：`network` / `local` / `database` / `system` / `fixed_sample` |
| `items[].data_sources[].url` | 真实 URL（network 类型时必填），G 必须用此 URL 发请求 |
| `items[].acceptance[]` | 验收标准数组，V 逐条检查是否实现 |
| `items[].migration_priority` | `high` / `medium` / `low`，决定实现顺序 |
| `items[].source_refs[]` | iOS 源码位置（文件:行号），便于追溯到原始 Swift 代码 |

→ **P** 按 feature 组织适配策略；**G** 按 feature id 领取实现任务；**V** 按 acceptance 逐条验证。

#### ★ screens.json — 页面规格（UI 生成的主要输入）

UI 信息来源优先级：

1. `SwiftSyntax` 解析 SwiftUI body，作为布局结构主证据，结果折叠进 `layout_spec` 和 `source_evidence`。
2. `XCUITest Accessibility Tree` 必须尝试读取模拟器运行时 UI 树，确认真实渲染出的文本、按钮、Tab、输入框、frame、enabled/selected 等结构化 UI 状态，结果或失败原因折叠进 `runtime_ui_tree_evidence`。
3. 源码核对跨文件状态、数据流、导航目标和控件 effect。
4. iOS 模拟器截图默认不采集；后续按需开启时，用于补充真实视觉效果和运行态分支，结果或失败原因折叠进 `screenshot_evidence` 和 `visual_notes`。

SwiftSyntax 负责回答“源码里写了什么结构”，XCUITest UI 树负责回答“运行时实际渲染了哪些可访问元素”。截图能力保留，用于按需回答“实际看起来是什么样”。模型负责把已采集证据归纳成可消费的同一份 `screens.json`。运行时 UI 树和截图都不能单独决定功能边界、数据来源、模块职责或平台能力。

| 字段 | 说明 |
|------|------|
| `items[].layout_confidence` | UI 置信度：`high` / `medium` / `low` |
| `items[].source_evidence[]` | SwiftSyntax / 源码证据，记录 View、文件和节点路径 |
| `items[].runtime_ui_tree_evidence[]` | XCUITest Accessibility Tree 证据，记录运行时元素、frame、label、identifier、状态 |
| `items[].screenshot_evidence[]` | 截图证据，记录截图路径、状态、补充的视觉信息和差异 |
| `items[].visual_notes[]` | 字体、颜色、间距、图片裁剪、空态/有数据态等运行态视觉补充 |
| `items[].layout_spec.container` | 容器层级，如 `NavigationStack > ScrollView > VStack` → ArkUI `Navigation > Scroll > Column` |
| `items[].layout_spec.sections[]` | 页面分区，每个 section 包含 elements 数组 |
| `items[].layout_spec.sections[].elements[].role` | 元素角色：`text` / `image` / `button` / `toggle` / `picker` / `input` 等 |
| `items[].layout_spec.sections[].elements[].effect` | **功能效果链路**：记录控件值被消费的位置和产生的行为变化 |
| `items[].layout_spec.navigation_bar.trailing_items[]` | 工具栏按钮（如 info / + 按钮），每个必须有 action 或 navigates_to |
| `items[].layout_spec.tab_bar.items[]` | Tab 栏定义（label + icon + target_screen），仅在根 TabView 页面出现 |
| `items[].navigates_to[]` | 导航出口数组：target + trigger + style（navigation_push / sheet / tab_switch） |
| `items[].interactions[]` | 交互数组：pull_to_refresh / swipe_action / toolbar / alert 等 |

→ **P** 从 navigates_to 聚合 navigation_map，从 interactions 提取交互映射；**G** 按 layout_spec 逐 section 生成 ArkUI 代码；**V** 逐 section 检查还原度。

XCUITest UI 树主要补充 SwiftSyntax 不擅长的运行态结构：

- 元素是否真实渲染出来，页面是否空白。
- 文本、按钮、Tab、输入框、列表项的运行时 label / value / identifier。
- 元素 frame、enabled、selected、存在性和可点击性。
- 条件分支实际走向，例如 `if recipes.isEmpty` 显示空态还是列表。
- 缺少 accessibilityIdentifier 时的稳定定位风险。

截图主要补充 SwiftSyntax 和 UI 树都不擅长的视觉信息：

- 真实字体层级、颜色、背景、圆角、阴影、间距。
- 图片资源是否正确显示、裁剪方式是否符合预期。
- 空态、有数据态、表单态、弹窗态、sheet 态。
- 系统控件真实样式，如 Tab、NavigationBar、Picker、Toggle。

如果 SwiftSyntax、运行时 UI 树和截图不一致，`screens.json` 必须同时保留三种证据：以源码结构作为功能事实，以 UI 树作为运行态结构事实，以截图作为视觉事实，并在 `visual_notes` 中标记差异。

#### functions.json — 函数索引

| 字段 | 说明 |
|------|------|
| `items[].kind` | `model` / `service` / `store` / `view_model` / `utility`，决定 Harmony 侧文件归属 |
| `items[].migration_action` | `model` / `service` / `store` / `arkui_component` / `platform_adapter` / `merge` / `delete_with_reason` |
| `items[].concurrency` | 并发模式枚举（11 种），G 查 platform-capabilities.json 映射为 ArkTS 并发原语 |
| `items[].concurrency_detail` | 并发参数（如 `maxConcurrency=5`、`interval=1s`、`delay=5s`） |
| `items[].type_definition` | 仅 `kind=model` 时出现：列出 enum cases（含 rawValue、显示名）或 struct fields |

→ **P** 提取 concurrency 做能力匹配；**G** 按 migration_action 决定文件归属，按 type_definition 生成模型；**V** 检查并发模式是否正确实现。

#### capabilities.json — iOS 系统能力

| 字段 | 说明 |
|------|------|
| `items[].id` | 能力 id（如 `cap-cloudkit`、`cap-userdefaults`） |
| `items[].capability` | iOS framework / API 名称 |
| `items[].runtime_behavior` | 运行时行为描述 |
| `items[].permission_or_entitlement` | 所需权限或 entitlement |
| `items[].feature_ids[]` | 关联的 feature id 列表 |

→ **P** 以此为输入做能力映射；**V** 检查每个能力是否有对应 Harmony 实现。

#### modules.json — 模块结构

| 字段 | 说明 |
|------|------|
| `items[].id` | 模块 id（如 `views.method`） |
| `items[].ios_paths[]` | iOS 源码目录/文件 |
| `items[].public_interfaces[]` | 对外接口或主要类型 |
| `items[].depends_on[]` / `used_by[]` | 模块间依赖关系 |
| `items[].suggested_harmony_boundary` | 建议的 Harmony 侧文件边界 |

→ **G** 按模块拆分 Harmony 文件结构；**P** 按模块边界组织适配策略。

#### resources.json — 资源清单

| 字段 | 说明 |
|------|------|
| `items[].type` | `image` / `color` / `sf_symbol` / `markdown` / `app_icon` |
| `items[].name` | 资源名（如 `AeroPress-Big`、`coffee`） |
| `items[].value` / `value_light` / `value_dark` | 颜色的 hex 值（含深色模式） |
| `items[].used_by_features[]` | 关联 feature id |

→ **P** 从 sf_symbol 类型提取 symbol_map；**G** 按颜色和图标资源生成 ArkUI 资源文件。

#### project.json — 工程元数据

| 字段 | 说明 |
|------|------|
| `bundle_identifier` | iOS bundle id → Harmony bundleName |
| `app_entry.file` / `app_entry.type` | 启动入口文件和类型 |
| `entitlements.capabilities[]` | entitlement 列表 |
| `imports[]` | 所有 import 的 framework（含第三方） |

→ **G** 用 bundle_identifier 设置 Harmony 工程配置。

**分析范围**：只分析主 App target。排除 Test、Widget、Extension、SnapshotSupport。

### Stage 2: platform-adaptation — 平台能力适配

**目标**：把 iOS 事实转换为 HarmonyOS 侧可执行的迁移策略。

**输入**：

| 输入 | 来源 |
|---|---|
| capabilities.json | ios-analyze |
| features.json | ios-analyze |
| modules.json | ios-analyze |
| functions.json | ios-analyze |
| screens.json | ios-analyze |
| resources.json | ios-analyze |
| platform-capabilities.json | 参考库（可扩展） |

**产出（5 个 JSON + 1 个 MD）**：

```
output_{{PROJECT_NAME}}/platform-adaptation/
├── capability-coverage.json     # 每个 iOS 能力 → HarmonyOS Kit 映射 + 策略
├── feature-adaptation.json      # 每个 feature 的适配方案
├── implementation-guidance.json # platform_modules 边界 + public API 草案
├── interaction-adaptation.json  # ★ 导航地图 + 交互映射 + SF Symbol 映射
├── risks.json                   # 风险清单（含 recommended_action）
└── reports/平台能力适配摘要.md
```

**核心产物详解**：

#### ★ feature-adaptation.json — 每 feature 的适配方案（G 必读）

告诉 G 每个 feature 需要创建什么 service/store/platform adapter，以及 UI 生成的约束。

| 字段 | 说明 |
|------|------|
| `features[].feature_id` | 关联 features.json 的 feature id |
| `features[].adaptation_items[].capability_id` | 命中的平台能力 |
| `features[].adaptation_items[].strategy` | 适配策略枚举 |
| `features[].adaptation_items[].generation_tasks[]` | **G 必须逐项实现的任务清单**（如 "Create PreferencesService wrapping @ohos.data.preferences"） |
| `features[].adaptation_items[].ui_constraints[]` | **G 生成 UI 时必须遵守的约束**（如 "Instagram button must open Web view in a bindSheet"） |
| `features[].adaptation_items[].implementation_boundary` | 目标文件路径（如 `entry/src/main/ets/services/PreferencesService.ets`） |
| `features[].adaptation_items[].required_permissions[]` | 所需权限列表 |
| `features[].adaptation_items[].fallback` | 降级方案 |

→ **G** 实现每个 feature 前必须读 generation_tasks[] 逐项实现，读 ui_constraints[] 在 UI 中遵守。**V** 检查 generation_tasks 是否全部实现。

#### ★ risks.json — 风险清单（G 必读）

记录高风险能力的迁移决策。G 实现涉及平台能力的代码前必须读此文件。

| 字段 | 说明 |
|------|------|
| `risks[].id` | 风险 id |
| `risks[].level` | `high` / `medium` / `low` |
| `risks[].capability_id` | 关联的能力 id |
| `risks[].problem` | 问题描述（如 "CloudKit 是 Apple 平台专有，HarmonyOS 无 SDK"） |
| `risks[].decision_needed` | 需要人工决策的问题 |
| `risks[].recommended_action` | **G 必须执行的建议方案**（high 级不允许跳过） |

→ **G** 实现涉及平台能力前读 recommended_action；**V** 维度 7c 检查每条 high 风险的 recommended_action 是否落实。

#### ★ interaction-adaptation.json — 桥接三表

| 子表 | 字段 | 说明 |
|------|------|------|
| `navigation_map[]` | `from` / `to` / `trigger` / `style` / `harmony_implementation` | 每条导航的 Harmony 实现（pushUrl / bindSheet / tab_switch） |
| `interactions[]` | `screen_id` / `type` / `ios_modifier` / `harmony_component` / `implementation_note` | 每个交互的组件映射（如 `.refreshable` → `Refresh`） |
| `symbol_map[]` | `ios_symbol` / `harmony_resource` / `mapping_type` / `usage` | 每个 SF Symbol → Harmony 图标（如 `plus` → `$r('sys.media.ohos_ic_public_add')`） |

→ **G** 按 navigation_map 生成路由注册和导航调用；按 interactions 选择 ArkUI 组件；按 symbol_map 使用图标。**V** 检查每条导航、交互、图标映射是否实现。

#### capability-coverage.json — 能力映射结果

| 字段 | 说明 |
|------|------|
| `items[].capability_id` | 能力 id |
| `items[].status` | `mapped` / `unmapped` |
| `items[].harmony.kit` / `harmony.api_area` | Harmony 侧 Kit 和 API |
| `items[].harmony.strategy` | 适配策略 |
| `unmapped[]` | 参考库未覆盖的能力 + 临时建议 |

→ **V** 维度 7b 检查每个 mapped 能力有实现，每个 unmapped 有决策。

#### implementation-guidance.json — 代码生成指导

| 字段 | 说明 |
|------|------|
| `platform_modules[].id` | 平台模块 id（如 `platform.preferences`） |
| `platform_modules[].target_path` | 目标文件路径 |
| `platform_modules[].public_api[]` | 公开 API 草案（G 按此生成接口签名） |
| `platform_modules[].implementation_tasks[]` | 实现任务清单 |
| `platform_modules[].verification[]` | 验证标准（V 检查这些标准是否满足） |
| `manifest_requirements[]` | manifest 配置需求 |
| `service_requirements[]` / `store_requirements[]` | 需要创建的 service 和 store |

→ **G** 按此文件创建 platform 层文件和 service/store；**V** 维度 7 检查每个 module 文件存在且有 public_api。

**策略枚举**：`native_equivalent` / `api_replacement` / `product_degradation` / `custom_service` / `unsupported_pending_decision`

### Stage 3: harmony-generate — Harmony 工程生成

**目标**：按 JSON 规格生成完整可构建的 HarmonyOS NEXT 工程。

**输入**：

| 输入 | 来源 |
|---|---|
| 7 个 JSON specs | ios-analyze |
| 5 个适配 JSON | platform-adaptation |
| project-template/ | 工程模板 |
| `screens.json` 中的 `layout_spec` / `runtime_ui_tree_evidence` / `screenshot_evidence` / `visual_notes` | ios-analyze |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-generate/
├── harmony模块实现计划.json    # 每个 iOS 模块 → Harmony 文件映射
└── harmony全量实现追踪.md      # 追踪表：iOS 类型/函数 → Harmony 去向 + 状态

{{PROJECT_NAME}}Harmony/         # 完整 HarmonyOS NEXT 工程
└── entry/src/main/ets/
    ├── pages/                   # 每个 screen → 一个页面
    ├── components/              # component_specs 展开
    ├── stores/                  # @ObservedV2 + @Trace 状态管理
    ├── services/                # 网络请求、数据解析、业务逻辑
    ├── platform/                # 平台适配层（通知、定位、TTS、分享等）
    └── models/                  # 数据模型、常量
```

#### harmony模块实现计划.json — 模块实现计划

| 字段 | 说明 |
|------|------|
| `modules[].ios_module_id` | iOS 模块 id（对应 modules.json） |
| `modules[].harmony_files[]` | 该模块产出的 Harmony 文件列表 |
| `modules[].ios_types[]` | 该模块包含的 iOS 类型/函数 |
| `modules[].migration_action` | 迁移动作（model / service / store / merge / delete） |
| `modules[].status` | 实现状态（pending / in_progress / completed） |

→ **V** 按此文件检查每个模块是否都有对应 Harmony 文件。

#### Harmony 工程目录职责

| 目录 | 来源 | 规则 |
|------|------|------|
| `pages/` | screens.json 每个 screen → 一个 `*Page.ets` | 页面按 layout_spec 生成，用 Navigation/router 管理导航 |
| `components/` | screens.json 的 component_specs | 可复用子组件，用 `@ComponentV2` + `@Builder` |
| `stores/` | functions.json `kind=store` 或 `view_model` | `@ObservedV2` + `@Trace`，单例模式 |
| `services/` | functions.json `kind=service` + implementation-guidance 的 service_requirements | 网络请求、数据持久化、业务逻辑 |
| `platform/` | implementation-guidance 的 platform_modules | 平台适配层（权限、通知、定位等） |
| `models/` | functions.json `kind=model` | 数据模型，含 enum type_definition 展开 |

**关键约束（部分，完整规则见 SKILL.md）**：

| 规则 | 说明 |
|---|---|
| ArkTS 严格模式 | 禁止 any/spread/解构/索引访问/barrel export |
| V2 装饰器 | @ComponentV2、@ObservedV2、@Trace、@Local、@Param |
| layout_spec 驱动 | UI 按 SwiftSyntax View Tree 整理出的结构化 JSON 树逐 section 生成，视觉补充来自 `screens.json.screenshot_evidence` / `visual_notes` |
| 消费 feature-adaptation | 每个 feature 实现前读 generation_tasks[] + ui_constraints[] |
| 消费 risks.json | 涉及平台能力前读 recommended_action，high 级不允许跳过 |
| 功能端到端实现 | 控件 → 存值 → 效果生效，三层必须全做 |
| 真实数据 | 使用与 iOS 相同的 URL，HTTP 错误用 hilog 记录 |
| errorMessage 渲染 | 数据失败时显示错误 + 重试，不允许空白页面 |
| SF Symbol 映射 | 读 symbol_map，图标 + 文字分两行（不拼字符串） |
| 导航完整 | 每条 navigates_to 有路由代码，不跳过中间页面 |
| 并发还原 | 读 functions.json concurrency 字段 + 查 platform-capabilities.json，按映射实现（timer→setInterval、dispatch_async→setTimeout 等） |
| 空态/错误态 | 每个 states 数组中的状态都有对应 UI |
| 设置效果 | 每个 toggle/picker 沿 iOS 源码追踪消费侧并实现 |

**目标**：对照 iOS 分析产物逐维度验证，产出 gaps 清单。

**输入**：

| 输入 | 来源 |
|---|---|
| features.json | ios-analyze |
| screens.json | ios-analyze |
| functions.json | ios-analyze |
| capabilities.json | ios-analyze |
| modules.json | ios-analyze |
| resources.json | ios-analyze |
| capability-coverage.json | platform-adaptation |
| feature-adaptation.json | platform-adaptation |
| interaction-adaptation.json | platform-adaptation |
| implementation-guidance.json | platform-adaptation |
| risks.json | platform-adaptation |
| harmony模块实现计划.json | harmony-generate |
| Harmony 工程 | harmony-generate |

**产出**：

```
output_{{PROJECT_NAME}}/harmony-verify/
├── verification-results.json    # 13 个维度的逐项验证结果
├── gaps.json                   # 缺口清单（含 suggested_fix）
└── reports/迁移验收摘要.md
```

#### verification-results.json — 验证结果

| 字段 | 说明 |
|------|------|
| `build.status` | 构建是否通过（passed / failed） |
| `summary.overall` | 总体结论：`passed` / `passed_with_gaps` / `failed` |
| `summary.total_features` / `passed_features` | 功能覆盖统计 |
| `features[].feature_id` / `status` / `acceptance_results[]` | 每个 feature 的逐条验收结果 |
| `navigations[]` / `interactions[]` / `concurrency[]` | 各维度逐项检查结果 |
| `capability_coverage[]` | 维度 7b：每个能力是否实现 |
| `risks[]` | 维度 7c：每条风险是否缓解（mitigated / not_mitigated） |

#### gaps.json — 缺口清单（喂回 G 修复）

| 字段 | 说明 |
|------|------|
| `gaps[].id` | gap 编号 |
| `gaps[].dimension` | 所属维度（feature_coverage / navigation / risk 等） |
| `gaps[].severity` | `critical` / `high` / `medium` / `low` |
| `gaps[].description` | 缺口描述 |
| `gaps[].ios_evidence` | iOS 侧的期望（来自 specs） |
| `gaps[].harmony_evidence` | Harmony 侧的现状（来自源码） |
| `gaps[].suggested_fix` | **具体修复方案**（含文件路径和代码改动建议） |

→ **G** 读 gaps.json 逐条修复，修完重新构建 + verify，直到 gaps 清零或仅剩 low 级。

**验证维度**：

| # | 维度 | 检查内容 | 通过标准 |
|---|---|---|---|
| 1 | 构建验证 | HAP 能否构建成功 | 必须 100% |
| 2 | 功能覆盖 | 每个 feature 有实现文件和实质代码 | ≥ 95% |
| 3 | 导航可达 | 每条 navigates_to 有路由代码 | 100% |
| 4 | 交互还原 | 每个 interaction 有 ArkUI 实现 | ≥ 90% |
| 5 | 并发还原 | functions.json 中标注的并发模式在代码中正确实现 | 100% |
| 6 | 真实数据 | URL 一致、有错误处理、有权限 | 100% |
| 7 | 平台能力 | 每个 platform module 文件存在且有 API + verification[] | ≥ 95% |
| 7b | 能力覆盖 | capability-coverage.json 每个 mapped 能力有实现，unmapped 有决策 | ≥ 95% |
| 7c | 风险缓解 | 每条 high 风险的 recommended_action 已落实 | 100% (high) |
| 8 | layout_spec 还原 | 每个 section 有代码、tab 有图标、picker 有 Select | ≥ 90% |
| 8a | 运行时 UI 树对齐 | XCUITest UI 树中的主要 text/button/tab/input/cell 在 Harmony 中可见且状态一致 | ≥ 90% |
| 8b | 快捷操作入口可达 | 每个 toolbar/导航按钮有对应 onClick 实现 | 100% |
| 8c | 功能效果链路 | 控件操作后 App 行为会变（toggle/picker/button → 效果） | 100% |
| 9 | 视觉还原 | `screens.json.screenshot_evidence` / `visual_notes` 与 layout_spec 对照 | 有截图证据时必须核对，缺失时标记视觉证据不足 |

**反馈循环**：gaps.json 喂回 harmony-generate 修复，修完再 verify，直到 gaps 清零。

```
harmony-generate ──→ harmony-verify ──→ gaps.json
      ↑                                        │
      └────────── 修复 gaps ←──────────────────┘
```

## 演进路线：Skill → 多 Agent

### 当前版本：Skill 模式

首版本用 4 个 skill 在同一上下文中顺序执行，验证流程可行性和产物质量。

**优点**：实现简单，调试方便，无需额外编排逻辑。

**局限**：

- 模型可能绕过 JSON specs 直接读 iOS 源码，产物契约不强制。
- 某阶段失败需要重跑全流程，无法定点重试。
- 全流程串行执行，Bean-Juice 约 60 分钟。
- 各阶段共享上下文窗口，前面阶段读的大量源码挤压后面阶段的可用空间。

### 后续版本：多 Agent 模式

每个阶段由独立 agent 承载，agent 之间只通过 JSON 文件交接。

**核心收益**：

| 收益 | 说明 |
|------|------|
| 强制产物契约 | Agent 只能看到显式传入的文件列表，无法偷看原始 iOS 源码，强迫只消费结构化 JSON |
| 隔离失败 + 定点重试 | verify 报 N 个 gap，只需重跑 G agent 修 gap，不重跑 ios-analyze 和 platform-adaptation |
| 并行加速 | harmony-generate 可按 feature 分拆到多个 agent 并行生成；ios-analyze 的各 step 也可并行 |
| 阶段间质量门 | Agent 之间插入 schema 校验脚本，features.json 不合格直接重跑，不等 G 才发现问题 |
| Token 预算隔离 | 每个阶段独立上下文空间，ios-analyze 读 40 个 Swift 文件不影响 G 的可用空间 |
| 差异化模型 | 分析阶段用强推理模型，生成阶段用执行型模型，控制成本 |

**架构示意**：

```
                    Agent 编排层
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
  ios-analyze     platform-        harmony-generate
    agent          adaptation         agent
        │           agent         ┌────┴────┐
        │               │        ▼         ▼
        ▼               ▼    feature-A   feature-B
   schema 校验      schema 校验   agent       agent
      (脚本)          (脚本)       │           │
        │               │         └────┬──────┘
        ▼               ▼              ▼
                   harmony-verify-agent
                          │
                     gaps.json
                          │
                   ┌──────┴──────┐
                   ▼             ▼
              gap < 阈值?    重跑 G agent
                   │           修复 gaps
                   ▼
                 完成
```

**切换条件**：当 skill 模式在 3+ 个不同工程上验证产物质量稳定后，切换为多 Agent 模式。
