# iOS 应用迁移到 HarmonyOS NEXT 方案

## 诉求

希望选择一个真实但规模可控的 iOS 工程，让模型完成从 iOS 工程理解到 HarmonyOS NEXT 工程实现的迁移流程。迁移不应直接从代码改写开始，而应先让模型完整阅读原工程，梳理功能、页面、数据流和平台能力，再基于清晰的规格文档实现 HarmonyOS NEXT 版本。

最终希望形成一套可复用的迁移方法：模型先产出设计文档和功能清单，再按清单逐步实现 HarmonyOS NEXT 应用。

### 样本应用要求

- 中小型应用，有真实业务结构和多个页面。
- 能在 iOS 模拟器真实运行主要流程。
- 包含 iOS 系统能力（通知、持久化、Widget、权限、设置等），需要在 HarmonyOS NEXT 中整改或替代。

### 验收重点

- 模型是否能准确理解原 iOS 工程，而不是只做表面代码翻译。
- 功能清单是否完整覆盖主要页面、数据流、状态和异常场景。
- HarmonyOS NEXT 实现是否保持主要设计和交互一致。
- iOS 平台能力是否被明确映射到 HarmonyOS NEXT Kit。
- 迁移结果是否能构建运行，并能完成核心业务流程。

## 样本工程：NewsMobile

`NewsMobile` 是一个 SwiftUI 新闻阅读应用：

- 主流程：新闻首页、推荐、搜索、收藏、设置、文章详情。
- 真实数据：公开 RSS、Open-Meteo 天气接口，不需要 API Key。
- Apple 原生能力较多：通知、定位、后台刷新、Widget、iCloud Sync、本地 API、TTS、NLP。
- 全量迁移目标：主 App、Widget/卡片、云同步、后台刷新、本地 API server 等都进入迁移范围。

## 迁移流程

```
iOS 工程分析 → 平台能力适配 → Harmony 工程生成 → 视觉验收
```

每个阶段对应一个独立 skill，通过 skill 直接调用。阶段之间通过本地文件交接，不通过对话上下文交接。

## Skill 设计

### 1. ios-analyze

分析 iOS 原工程事实，输出结构化 JSON specs。

输入：iOS 工程目录

输出：
- `output/ios-analyze/specs/*.json` — project、modules、features、functions、screens、capabilities、resources（7 个 JSON）
- `output/ios-analyze/reports/*.md` — 人工摘要
- `output/ios-analyze/screenshots/png/` — 截图（如采集）

核心产物 `features.json`：三级功能结构，每个 feature 有稳定 id、source_refs、acceptance。JSON 之间通过稳定 id 互相引用。

### 2. platform-adaptation

把 iOS 平台能力转换为 HarmonyOS NEXT 侧可执行的迁移策略。

输入：
- `output/ios-analyze/specs/capabilities.json`
- `output/ios-analyze/specs/features.json`
- `output/ios-analyze/specs/modules.json`
- `output/ios-analyze/specs/functions.json`
- `skills/platform-adaptation/references/platform-capabilities.json`（可扩展参考库）

输出：
- `output/platform-adaptation/capability-coverage.json` — 命中能力集合
- `output/platform-adaptation/feature-adaptation.json` — feature 级适配策略
- `output/platform-adaptation/implementation-guidance.json` — 代码生成指导（含 platform_modules、manifest、权限）
- `output/platform-adaptation/risks.json` — 可执行风险清单

策略枚举：`native_equivalent` / `api_replacement` / `product_degradation` / `custom_service` / `unsupported_pending_decision`

不纳入：基础 UI 控件映射（Button/Text/List/TabView 等），这些属于 harmony-generate 的 ArkTS 编码规范。

### 3. harmony-generate

基于 JSON specs、平台适配产物和工程模板，生成完整 HarmonyOS NEXT ArkTS/ArkUI 工程。

输入：
- `output/ios-analyze/specs/*.json`（7 个 JSON）
- `output/platform-adaptation/*.json`（4 个 JSON）
- `skills/harmony-generate/references/project-template/`（HarmonyOS NEXT 空工程模板）
- 截图产物

输出：
- 完整 HarmonyOS NEXT 工程
- `output/harmony-generate/harmony模块实现计划.json` — 按模块拆分的实现计划
- `output/harmony-generate/harmony全量实现追踪.md` — 每个 iOS 类型/函数的 Harmony 去向

关键约束：
- 必须严格遵守 ArkTS 严格模式编码规范（见下方）
- 必须按功能清单全量实现
- Store 必须用 `@ObservedV2 + @Trace` 实现响应式
- 装饰器必须用 V2 版本

### 4. harmony-verify

对照 iOS 分析产物（功能清单、导航图、交互列表、数据源）逐项验证 Harmony 工程的迁移完整性。

## ArkTS 编码规范（harmony-generate 强制）

HarmonyOS NEXT 的 ArkTS 使用严格模式，以下语法禁止使用：

1. 禁止 `any`、`unknown` 类型
2. 禁止对象展开 `{...obj}` — 逐字段赋值
3. 禁止解构声明 `const {a, b} = obj`
4. 禁止索引访问 `obj[key]`
5. 禁止对象字面量作类型声明 `Array<{...}>` — 先声明 interface
6. 禁止无类型对象字面量 — 用 `as Type`
7. 禁止函数隐式返回类型 — 显式标注
8. 装饰器用 V2：`@ComponentV2`、`@ObservedV2`、`@Trace`、`@Local`、`@Param`、`@Provider`、`@Consumer`、`@Event`
9. Store 必须用 `@ObservedV2 + @Trace`（SwiftUI `ObservableObject + @Published` 的等价机制）
10. 页面组件用 `@Local` 持有 Store
11. 不允许 barrel export（index.ets）— 直接 import 具体文件路径
12. RegExp 是独立类型，不能赋值给 string

## 平台能力映射参考

SwiftUI 响应式框架到 ArkUI V2 的映射：

| iOS (SwiftUI) | HarmonyOS (ArkUI V2) | 用途 |
|---|---|---|
| `ObservableObject` | `@ObservedV2` | 可观察的 class |
| `@Published` | `@Trace` | 属性变化追踪 |
| `@StateObject` | `@Local` | 组件内创建可观察对象 |
| `@ObservedObject` | `@Param` | 父传子 |
| `@EnvironmentObject` | `@Provider` + `@Consumer` | 跨层级共享 |
| `@State` | `@Local` | 组件内部状态 |
| `@Binding` | `@Param` + `@Event` | 双向绑定拆为属性+回调 |

其他平台能力映射见 `skills/platform-adaptation/references/platform-capabilities.json`。

## 验收标准

iOS 侧：
- 首页能看到新闻数据或截图模式 固定样例数据。
- 自动截图覆盖首页、分类、详情、搜索、收藏、设置。
- 能输出完整 JSON specs。

HarmonyOS NEXT 侧：
- 应用能构建运行。
- 首屏有新闻内容，不出现空白页。
- 主 Tab、列表、详情、搜索、收藏、设置可用。
- WebView、TTS、通知、定位、卡片、云同步、后台刷新、本地 API 等能力均有实现或适配入口。
- UI 能对照 iOS 截图说明复刻关系。
- 每个 Harmony 实现都能追溯到 features.json、screens.json 或 platform-adaptation JSON。
