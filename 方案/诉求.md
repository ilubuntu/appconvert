# iOS 应用迁移到 HarmonyOS NEXT 的诉求

## 目标

希望选择一个真实但规模可控的 iOS 工程，让模型完成从 iOS 工程理解到 HarmonyOS NEXT 工程实现的迁移流程。迁移不应直接从代码改写开始，而应先让模型完整阅读原工程，梳理功能、页面、数据流和平台能力，再基于清晰的规格文档实现 HarmonyOS NEXT 版本。

最终希望形成一套可复用的迁移方法：模型先产出设计文档和功能清单，再按清单逐步实现 HarmonyOS NEXT 应用，并通过截图、交互说明和能力映射保证迁移结果接近原 iOS 应用。

## 样本应用要求

- 不能太简单。天气类 App 功能和页面过少，不足以验证模型对真实工程的迁移能力。
- 不能太复杂。类似 IceCubesApp 这种社交/IM 类工程包含大量社交流、私信、推送服务端和多扩展能力，首轮迁移成本过高。
- 应该是中小型应用，但要有真实业务结构和多个页面。
- 应能在 iOS 模拟器真实运行主要流程，尽量避免必须依赖实体 iPhone、Apple Developer 证书或 APNs 远程推送配置的功能。
- 可以包含部分 iOS 系统能力，例如登录、本地通知、本地数据持久化、Widget/小组件、权限、设置等，这些能力需要在 HarmonyOS NEXT 中整改或替代实现。

## 期望迁移流程

1. 模型读取 iOS 工程完整代码和配置。
2. 模型输出 iOS 原工程审计文档，说明工程结构、入口、模块、依赖和平台能力。
3. 模型输出功能清单，覆盖主要页面、用户流程、数据模型、状态变化和异常场景。
4. 模型结合主要界面截图输出页面规格，尽量保持 HarmonyOS NEXT 版本的设计、交互和信息层级与原应用一致。
5. 模型输出 iOS 能力到 HarmonyOS NEXT Kit 能力的映射表。
6. 模型按规格文档逐步实现 HarmonyOS NEXT 工程。
7. 最后通过构建、手测和关键功能对比验收迁移结果。

## 文档产物要求

迁移前应优先生成以下文档：

- `source-audit.md`：iOS 工程结构、target、依赖、入口、模块和平台能力审计。
- `module-spec.md`：按 iOS 项目结构梳理模块职责、对外接口、模块依赖关系和 HarmonyOS NEXT 模块拆分参考。
- `feature-spec.md`：功能清单、页面清单、核心用户流程和异常状态。
- `screen-spec.md`：核心页面截图、布局结构、交互行为和视觉要求。
- `api-spec.md`：后端 API、请求/响应模型、鉴权流程和错误处理。
- `capability-map.md`：iOS Apple SDK、Capabilities、Entitlements 到 HarmonyOS NEXT Kit 的映射。
- `migration-plan.md`：全量迁移范围、等价替代策略、外部配套要求、实现顺序和验收标准。

## Skill 诉求

迁移流程不应由一个大而全的 skill 处理，而应拆成五个职责清晰的 skill：

- `ios-analyze`：分析 iOS 原工程事实，输出结构化 JSON specs（project.json、modules.json、features.json、functions.json、screens.json、capabilities.json、resources.json）、人工摘要和辅助截图。
- `platform-adaptation`：基于 JSON specs 和平台能力参考库，输出可执行的适配策略（capability-coverage.json、feature-adaptation.json、implementation-guidance.json、risks.json），明确 native_equivalent / api_replacement / product_degradation / custom_service / unsupported_pending_decision。
- `harmony-generate`：基于 JSON specs、平台适配产物和截图辅助生成 HarmonyOS NEXT 工程。
- `harmony-visual-verify`：对照 iOS 截图逐页验收和修复 HarmonyOS NEXT 迁移工程的 UX 与 UI。
- `build-harmony-project`：HarmonyOS NEXT 工程构建能力。

`ios-analyze` 应包含：

- iOS 工程审计流程。
- JSON specs 输出模板（project.json、modules.json、features.json、functions.json、screens.json、capabilities.json、resources.json）。
- 按 iOS 项目结构输出模块职责、对外接口、模块依赖关系和 HarmonyOS NEXT 参考拆分。
- iOS Apple SDK、Capabilities、Entitlements、App Extensions 的识别方法。
- 自动化截图流程，禁止人工点击模拟器采图。
- 一个扫描脚本，用于读取 `.xcodeproj`、`Info.plist`、entitlements、SPM 依赖和 Swift imports，生成初始能力清单。
- 三级功能结构（一级→二级→三级），每个 feature 有稳定 id、source_refs、acceptance。
- JSON 之间通过稳定 id 互相引用。
- 人工摘要（Markdown）从 JSON specs 派生，不作为唯一事实源。

`platform-adaptation` 应包含：

- iOS 能力到 HarmonyOS NEXT Kit 的映射和推荐实现方式。
- 可扩展的平台能力参考库（`platform-capabilities.json`）。
- 策略枚举：native_equivalent / api_replacement / product_degradation / custom_service / unsupported_pending_decision。
- feature 级适配策略，含 implementation_boundary、public_api 草案、权限和 manifest 需求。
- 可执行风险清单，每条风险有 recommended_action。
- 不做基础 UI 控件映射（SwiftUI View/Button/List/TabView 到 ArkUI）。

`harmony-generate` 应包含：

- HarmonyOS NEXT 工程结构生成规则。
- ArkTS/ArkUI 页面、数据模型和 service 生成规则。
- 按 `implementation-guidance.json` 的 platform_modules 实现平台适配层。
- 构建验证和迁移缺口记录流程。

`harmony-visual-verify` 应包含：

- iOS/Harmony 截图逐页对比规则。
- 信息层级、布局、字号、颜色、图标、间距、状态和交互的对比维度。
- ArkUI 修复规则和界面对齐报告格式。

## 验收重点

- 模型是否能准确理解原 iOS 工程，而不是只做表面代码翻译。
- 功能清单是否完整覆盖主要页面、数据流、状态和异常场景。
- HarmonyOS NEXT 实现是否保持主要设计和交互一致。
- iOS 平台能力是否被明确映射到 HarmonyOS NEXT Kit，而不是被忽略。
- 迁移结果是否能构建运行，并能完成核心业务流程。
